# 實驗 4：SLUBStick 時間側通道

## 目的

展示如何用**分配時間**偵測 kernel slab allocator 何時建立新的 slab page，
識別哪些物件在同一個 slab page 上。

這是論文引用的 SLUBStick 技術的核心，也是 Allocator Massaging（實驗 5）的基礎。

---

## 原理

Slab allocator 分配物件時有兩種情況，時間差異明顯：

```
情況 A（快）：目前的 slab page 還有空位
  → 直接從空位清單取出一個 slot
  → 約 1000～2000 cycles

情況 B（慢）：目前的 slab page 已滿
  → 向 Page Allocator 要一個新的 4kB 頁面
  → 把新頁面切成固定大小的 slot
  → 才返回第一個 slot
  → 約 50000～300000 cycles
```

每個 slab page 的 slot 數量固定，所以「慢分配」會以**固定週期**出現：

```
時間序列：快 快 快 ... 快 [慢] 快 快 快 ... 快 [慢] 快 快 ...
           ←── 32 個 ──→       ←── 32 個 ──→
                                ↑
                       kmalloc-cg-128：4096 ÷ 128 = 32 個 slot
```

觀察到這個週期，就能確認每個 slab page 有幾個物件。

---

## msg_msg 的 slab cache

實驗使用 `msg_msg` 作為目標物件：

```
msg_msg header：約 48 bytes
+ payload：         72 bytes
= 總計：           120 bytes  →  kmalloc-cg-128（128 bytes 的 cache）

每個 slab page：4096 ÷ 128 = 32 個 slot
```

## 為什麼用「中位數 × 5」當閾值？

```
快分配時間：約 1000～3000 cycles（分布集中）
慢分配時間：約 50000～300000 cycles（遠大於快分配）

兩者之間有一個清楚的空白帶，
中位數代表「典型的快分配時間」，
× 5 確保不把快分配誤判為慢分配，
也不把真正的慢分配漏掉。
```

若環境噪音大（如 VM），可調整倍數（如 × 3 或 × 10）。

## 與論文的關係

SLUBStick 是 Maar et al.（2024）提出的技術，論文在此引用它來：

1. **驗證 massaging 成功**：確認目標 slab page 只包含我們要的物件
2. **識別同一 slab page 的物件**：把「慢」之後的 32 個快分配歸為同一頁
3. **為 TLB 攻擊做準備**：TLB 洩漏頁面位址後，slab page 內每個物件的位址可直接計算

```
slab_base + 128 × n   （n = 0, 1, ..., 31）
```
